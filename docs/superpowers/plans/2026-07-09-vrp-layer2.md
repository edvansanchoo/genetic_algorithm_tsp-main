# Camada 2 — VRP hospitalar Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Extend the Camada 1 mesh TSP app into a multi-vehicle VRP with a separate depot, capacity-limited trips that return to depot, greedy fixed assignment, and per-vehicle genetic search.

**Architecture:** Keep one `main.py` / Pygame loop. Add VRP models, greedy assignment, capacity decoder, and N independent GA populations (one per vehicle). Distances stay on the Camada 1 mesh (depot included as a graph node). UI shows depot, per-vehicle colored routes, fleet/capacity sliders, and N convergence series.

**Tech Stack:** Python 3.9+, Pygame, Matplotlib, unittest; reuse `order_crossover`, `apply_mutation`, `evolve_next_generation` patterns, `delivery_mesh` / `road_network`.

## Global Constraints

- Branch: `feature/road-network-blocked-nodes` (continue Camada 1 work)
- **Do not create git commits** unless the user explicitly asks
- No fuel/autonomy (Camada 3)
- No global chromosome that moves deliveries between vehicles during evolution
- Depot is separate from deliveries; every trip is `DEPOT → … → DEPOT`
- Blocked nodes remain out of the graph; path failures → `∞`
- Prefer reusing Camada 1 mesh APIs over rewriting BFS

### Task 2: Capacity decoder + mesh distance + priority

**Files:**
- Create: `traveling_salesman_problem/problem/vrp_decoder.py`
- Test: `tests/test_vrp_decoder.py`

**Interfaces:**
- Consumes: `DeliveryMesh`, `delivery_segment_distance`, `expand` helpers; `DeliveryToken`
- Produces:
```python
@dataclass
class TripStop:
    node_id: str            # DEPOT_ID or delivery point id
    quantity: int           # 0 at depot
    coordinate: Coordinate

@dataclass
class Trip:
    stops: List[TripStop]   # starts and ends with DEPOT
    distance: float

@dataclass
class DecodedVehiclePlan:
    trips: List[Trip]
    total_distance: float
    priority_penalty: float  # Σ priority × visit_order (deliveries only, order across trips)
    fitness: float           # total_distance + weight * priority_penalty (or inf)

def decode_vehicle_permutation(
    tokens: List[DeliveryToken],
    permutation: List[DeliveryToken],  # same multiset as tokens
    depot: Coordinate,
    mesh: DeliveryMesh,
    capacity: int,
    priority_weight: float = 0.0,
) -> DecodedVehiclePlan
```

**Decoder rules (exact):**
1. Validate permutation is a rearrangement of `tokens` (by `(point_id, quantity)` multiset); else treat as invalid → `inf`.
2. `current_load = 0`; open trip with depot stop; `visit_order = 0`.
3. For each token in permutation:
   - If `current_load + quantity > capacity`: append path back to depot, close trip, start new trip at depot, `current_load = 0`.
   - Append path depot/last → delivery on mesh; add stop; `current_load += quantity`; `visit_order += 1`; add `priority * visit_order` to penalty.
4. After loop, return to depot; sum trip distances.
5. Any missing mesh path → `total_distance = fitness = inf`.

Use a hand-built `delivery_mesh_from_parts` network in tests (include `DEPOT` node).
